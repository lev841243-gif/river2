### Про подключение к БД — важно

База **закрыта от интернета**. Публичный хост `igkibibaxis.beget.app` из панели Beget
**не работает** даже с самого VPS — соединение отбивается. Подключение только
по приватной сети:

```
DATABASE_URL="postgresql://cloud_user:ПАРОЛЬ@10.16.0.1:5432/default_db"
```

Приватную сеть нужно включить **у обоих** сервисов (кнопка «Подключить» у строки

## ⚠️ Требование к PostgreSQL — проверить ДО деплоя

**Нужен PostgreSQL 13 или выше.**

Защита от двойного бронирования — это exclusion-констрейнт в БД, ему нужно расширение `btree_gist`. Миграция ставит его сама (`CREATE EXTENSION IF NOT EXISTS btree_gist`).

- **PG 13+** — `btree_gist` помечен как *trusted*, ставится владельцем базы **без прав суперюзера**. Проверено: миграции полностью проходят от обычного пользователя-владельца.
- **PG 12 и старше** — `btree_gist` требует суперюзера. `npx prisma migrate deploy` упадёт на второй миграции. Тогда: попросить поддержку Beget выполнить `CREATE EXTENSION btree_gist;` один раз в вашей базе — после этого миграции пройдут.

Проверить версию и доступность расширения (в веб-консоли, подставив свои данные):

```bash
psql "postgresql://ПОЛЬЗОВАТЕЛЬ:ПАРОЛЬ@ХОСТ:5432/ИМЯ_БД" -c "SHOW server_version;"
psql "postgresql://ПОЛЬЗОВАТЕЛЬ:ПАРОЛЬ@ХОСТ:5432/ИМЯ_БД" -c "SELECT name FROM pg_available_extensions WHERE name='btree_gist';"
```

Вторая команда должна вернуть `btree_gist`. Если пусто — расширение не установлено на сервере, нужна поддержка Beget.

Пользователь БД должен быть **владельцем** базы (так и будет, если создавали её через панель).

### Админка

```env
ADMIN_SESSION_SECRET="сгенерировать, 32+ символа"
```

Сгенерировать: `node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"`.
Этим секретом подписывается кука сессии. **Без него вход падает намеренно** — тихий
фолбэк на константу означал бы, что куку подделает кто угодно. Смена секрета
разлогинивает всех разом: таблицы сессий нет, инвалидировать больше нечем.

Самих админов в `.env` нет — они в таблице `Admin`. Регистрации в панели нет
(публичный адрес: аккаунт завёл бы кто угодно), поэтому после миграций:

```bash
cd /root/dno
npx tsx scripts/admin.ts add ivan@example.com 'Иван'   # пароль спросит отдельно
npx tsx scripts/admin.ts list
```

Пароль в аргументах **не передаётся** — скрипт спросит его сам, скрытым вводом,
и переспросит для проверки. Так пароль не оседает в истории команд и его нельзя
случайно скопировать вместе с готовой командой. Это не теория: на боевом сервере
уже завелись админ с адресом `ваш@email.ru` и админ с паролем `новый-пароль` —
оба раза пример из инструкции вставили целиком.

Адрес — только латиницей: форма входа проверяет его тем же правилом
(`adminEmailSchema`), и под кириллическим адресом войти невозможно.

Если терминал не даёт ввести пароль (запуск не из консоли), есть запасной путь —
bash спрашивает пароль сам, а скрипт берёт его из переменной:

```bash
read -s -p "Пароль: " P && echo && ADMIN_NEW_PASSWORD="$P" npx tsx scripts/admin.ts add ivan@example.com 'Иван'; unset P
```

Сменить пароль: `npx tsx scripts/admin.ts passwd ivan@example.com`.

## 8. Nginx + SSL

Готовый конфиг — `deploy/nginx-dno.conf`. Установка:

```bash
cp deploy/nginx-dno.conf /etc/nginx/sites-available/dno
ln -sf /etc/nginx/sites-available/dno /etc/nginx/sites-enabled/dno
rm -f /etc/nginx/sites-enabled/default
nginx -t && systemctl reload nginx
certbot --nginx -d new.prokatkaterov.ru --agree-tos --redirect -m ПОЧТА
```

### ⚠️ X-Forwarded-For — не используйте `$proxy_add_x_forwarded_for`

Очевидный вариант `proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;`
**дописывает** к заголовку, который прислал клиент. Код берёт первый адрес из
списка — а он оказывается тем, что клиент выдумал сам. Итог: рейт-лимит
обходится подстановкой фальшивого IP в каждом запросе.

Проверено атакой на живом сервере: с `X-Forwarded-For: 1.2.3.1` забаненный
клиент снова проходил. Правильно (и так в конфиге):

```nginx
proxy_set_header X-Forwarded-For $remote_addr;
```

Мы единственный прокси, поэтому реальный адрес всегда в `$remote_addr`, а
клиентский заголовок надо затирать, а не дополнять.

## 10. Обновления сайта — одной командой

Со своей машины:

```bash
ssh root@85.198.68.244 'bash /root/dno/scripts/deploy.sh'
```

Скрипт делает всё сам: `git pull`, `npm ci` (он же тянет `prisma generate` через
postinstall), `prisma migrate deploy`, сборку, подмену и перезапуск. В конце
проверяет, что сайт отвечает 200, **и сам откатывается, если нет**.

### Почему не серией команд руками

- **Порядок легко перепутать**, а цена теперь — боевой сайт. Забытый
  `prisma generate` после правки схемы = падение в рантайме, на этом уже наступали.
- **Прежняя процедура делала `rm -rf .next` у работающего сайта.** Next читает
  чанки с диска на каждый запрос, поэтому пересборка «на месте» ломала живой
  процесс: посетители получали ошибки всё время сборки — десятки секунд, а не пару
  секунд перезапуска. На staging это не замечали. Теперь сборка идёт в
  `.next-build` (`distDir` из `NEXT_DIST_DIR`), а подмена — одним `mv`, мгновенно.

### Простой при выкатке

Остаётся только перезапуск PM2 — секунды. Полностью убрать его мешает `fork_mode`:
процесс один. `cluster` с двумя инстансами дал бы выкатку без простоя, но рейт-лимит
живёт в памяти процесса (см. долги) и разъехался бы по инстансам.

### Откат вручную

Скрипт откатывается сам, но если нужно вернуться позже:

```bash
cd /root/dno
mv .next .next-bad && mv .next-old .next && pm2 reload dno --update-env
```

### ⚠️ Сборка проверяет типы — и это ново

В `next.config.mjs` было `ignoreBuildErrors: true` (осталось от скаффолда v0):
`next build` печатал «Skipping validation of types» и выдавал
«✓ Compiled successfully» на сломанном коде. Единственной защитой был отдельный
`tsc --noEmit`, про который легко забыть. Теперь `ignoreBuildErrors: false` —
сборка падает на ошибке типов, и сломанный код до клиентов не доезжает.
Проверено намеренной ошибкой: сборка вернула `Type error` и код выхода 1.

### ⚠️ Про `prisma generate` — грабли, на которых уже наступили

`migrate deploy` добавляет столбец в **базу**, но не обновляет сгенерированный
**клиент** в `node_modules`. Если пропустить `prisma generate`, приложение
соберётся и запустится, а в рантайме упадёт:

```
Unknown argument `tgMessageId`
```

Обычно `generate` срабатывает сам через `postinstall` при `npm ci`. Но если
зависимости не менялись и `npm ci` пропущен — клиент останется старым.
**Менялась `schema.prisma` → запускай `prisma generate`.**

### Про `.env` и переменные

`pm2 reload dno --update-env` перечитывает окружение. После правки `.env`
(токены, пароли) перезапуск обязателен — иначе процесс работает со старыми
значениями.

## 11. Переключение основного домена — СДЕЛАНО

Выполнено. Ниже — как это прошло на самом деле, чтобы повторить или откатить.

**Панель Beget → DNS → зона `prokatkaterov.ru`.** Подзоны правятся по отдельности:
`@` и `www` — это ДВА разных диалога. Правка только `@` оставила `www` на старом
сайте, и Let's Encrypt отказался выдавать на него сертификат — домен вёл не к нам.

| Подзона | Было | Стало |
|---|---|---|
| `@` (A) | `45.130.41.20` | `85.198.68.244` |
| `www` (A) | `45.130.41.20` | `85.198.68.244` |

Не трогать: `MX` и `TXT` (почта на Яндексе), `autoconfig`/`autodiscover` (тоже почта),
`new` (на нём был вебхук бота).

**Дальше на сервере:**

```bash
# 1. Добавить домены в server_name обоих блоков /etc/nginx/sites-available/dno
# 2. Один сертификат на все три имени — не выпускать отдельный, иначе
#    certbot перепишет ssl_certificate и поддомен останется без сертификата:
certbot --nginx --cert-name new.prokatkaterov.ru \
  -d new.prokatkaterov.ru -d prokatkaterov.ru -d www.prokatkaterov.ru \
  --expand --redirect
# 3. Вебхук бота — на основной домен:
curl -X POST "https://api.telegram.org/bot$TELEGRAM_BOT_TOKEN/setWebhook" \
  -d "url=https://prokatkaterov.ru/api/telegram/webhook" \
  -d "secret_token=$TELEGRAM_WEBHOOK_SECRET"
# 4. Проверить, что продление не сломано редиректом:
certbot renew --dry-run
```

**Порядок важен: сертификат выпускается ПОСЛЕ смены DNS.** Let's Encrypt проверяет
домен обращением к нему — пока он ведёт на старый сервер, выпуск невозможен. Отсюда
короткое окно (минуты), когда домен уже наш, а сертификата ещё нет: посетители видят
предупреждение безопасности. Поэтому DNS меняют и **сразу** зовут того, кто выпустит
сертификат. HSTS на старом сайте не было — иначе это была бы не «предупреждение», а
глухая блокировка без возможности продолжить.

**Про TTL.** У A-записей он 600 секунд, а не сутки: переключение и откат — минуты.
Значение из `nslookup -debug` брать нельзя, там TTL другой записи; спрашивать
авторитетный сервер: `Resolve-DnsName prokatkaterov.ru -Type A -Server ns1.beget.ru`.

**Пересборка не нужна:** приложение не привязано к домену (`NEXT_PUBLIC_SITE_URL`
в коде не используется, sitemap/canonical нет).

**Откат:** вернуть A-записи `@` и `www` на `45.130.41.20`. Старый сайт не тронут и
жив. Через ~10 минут всё как было.

**Старый сайт** оставить включённым минимум на сутки после переключения: часть
посетителей идёт по кэшу DNS. Потом скачать бэкап и убрать в архив.
