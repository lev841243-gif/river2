# Состояние проекта — «Судоходная Компания Дно»

Краткий хэндовер, чтобы продолжить разработку в любом режиме без потери контекста.

## Что это
Сайт аренды катеров в Санкт-Петербурге. Next.js 16 (App Router), React 19, TypeScript, Tailwind v4.
Двуязычный: RU на `/`, EN на `/en` (словарь — `lib/i18n.ts`). Домен заказчика: prokatkaterov.ru.
Бренд в интерфейсе: «Судоходная Компания Дно».

## Стек (согласован)
- Приложение: Next.js на **Beget** (VPS/Node, PM2 + Nginx).
- БД: **PostgreSQL** (Beget Cloud) + **Prisma** ORM.
- Авторизация админки: своя (Auth.js/сессия) — этап админки.
- Фото лодок: WebP в `public/boats/<папка>` (в репозитории). Для галереи «Моменты»
  вопрос хранилища решён в пользу папки `uploads/` в корне проекта (вне
  `public/`) — см. раздел «Галерея «Моменты»».
- Telegram (grammY) и Google Sheets — этапы уведомлений/отчётности.
- Хостинг и деплой — см. `Docs/DEPLOY_BEGET.md`.

## Готово (этап 1 — бронирование на сайте с календарём занятости)
Клиент бронирует полностью на сайте: катер → календарь → время «с»/«до» → контакты. Занятость проверяется в реальном времени.

**Правила бронирования — `lib/booking-rules.ts` (единый источник для клиента и сервера).**
Актуальные значения — в разделе «Правила бронирования — ПЕРЕСМОТРЕНЫ» ниже:
расписания нет, 1 ч – 7 суток, буфера нет, слот держит только CONFIRMED,
горизонт 180 дней, шаг сетки 30 мин. Меняются константами в одном файле.

**Время.** `lib/spb-time.ts` — все даты считаются явно в петербургском поясе (UTC+3, фикс. с 2014 г.), а не в локальном поясе браузера. Иначе гость из другого пояса на EN-версии забронировал бы не то время. Проверено под TZ Москвы/Нью-Йорка/Токио — результат одинаковый.

**Защита от двойного бронирования — на уровне БД.** Миграция `20260715183600_booking_overlap_guard`: exclusion-констрейнт `EXCLUDE USING gist` по `(boatId, tstzrange(startAt, endAt))` для блокирующих статусов. Проверка «посмотрели → вставили» не спасает от гонки двух одновременных заявок; констрейнт делает пересечение физически невозможным. Postgres отдаёт код `23P01`, `lib/bookings-db.ts` ловит его и превращает в понятный 409 `BOAT_BUSY`. Требует `btree_gist` (миграция ставит сама).

**Схема.** `startAt`/`endAt` — `@db.Timestamptz(3)`, `endAt` обязателен. Оба обязательны для констрейнта: от `timestamp` без таймзоны `tstzrange` зависит от сессионной TimeZone и не годится в индексное выражение. Прочие DateTime тоже переведены в timestamptz (как в ТЗ).

**Файлы:**
- `lib/booking-rules.ts` — правила + Zod-схема заявки + honeypot.
- `lib/spb-time.ts` — петербургское время.
- `lib/booking-slots.ts` — расчёт свободных слотов (общий для календаря и сервера).
- `lib/bookings-db.ts` — занятость, создание заявки, определение источника по UTM/referer, клиент по телефону.
- `lib/rate-limit.ts` — лимит частоты.
- `app/api/availability/route.ts` — `GET ?boatSlug&from&to` → занятые интервалы.
- `app/api/bookings/route.ts` — `POST`, Zod + honeypot + рейт-лимит (5/10 мин) + проверка занятости.
- `components/nevsky/booking-calendar.tsx` — календарь, занятые дни недоступны.
- `components/nevsky/booking-modal.tsx` — шаги «когда» → «контакты» → успех.
- `components/nevsky/booking-context.tsx` — `BookingProvider`/`useBooking`.
- `lib/utm.ts` + `components/nevsky/utm-capture.tsx` — UTM → cookie `utm_data` (1 год) → в заявку.

**Важно про `boatId`.** В UI лодка — это `slug` (`getBoats()` мапит `id: r.slug`), а `Booking.boatId` — FK на `Boat.id` (uuid). API принимает slug и резолвит его сам. Не слать uuid с клиента.

**Связь с менеджером.** Telegram/WhatsApp/звонок — только контакт, не канал брони (`contacts` в `lib/i18n.ts`, блок в `Cta`).

**Что проверено вживую** (локальная БД `river2_dev`): констрейнт отбивает пересечение и пропускает встык/отменённые; POST даёт 201, повтор — 409; прошедшее время, рейт-лимит, honeypot (молча 201 без записи), неизвестная лодка — 404; заявка через UI легла в БД с верным временем МСК, ценой (3 ч × 50 000 = 150 000) и историей статусов; календарь на RU и EN показывает занятость. `npx tsc --noEmit` и `npm run build` чисты.
Проверки буфера, минимума 2 ч и часов работы были в первой редакции правил — эти
ограничения потом отменены заказчиком, проверять там больше нечего.

## Готово — Telegram-уведомления
Заявка приходит в группу Telegram карточкой с кнопками. Пробел «менеджер не видит
заявок» закрыт.

**Где живёт бота:** отдельного процесса **нет**. Это два обработчика внутри самого
сайта — `lib/telegram.ts` (API + вёрстка сообщения) и
`app/api/telegram/webhook/route.ts` (нажатия). Крутится в том же PM2-процессе `dno`,
выкатывается вместе с сайтом. Работает через **вебхук**, а не polling: Telegram сам
стучится на `https://new.prokatkaterov.ru/api/telegram/webhook`.

- **Бот:** @river_dno_bot. **Группа:** «neva», chat_id `-5416776321`.
- **Кнопки:** WhatsApp и Telegram — прямые ссылки; «📋 Номер» копирует телефон;
  «Подтвердить» / «Отменить» / «Изменить время» / «Вернуть в работу».
- **Почему звонок через копирование:** проверено на живом боте — `tel:` в кнопке
  Telegram **отклоняет** («Wrong port number specified»), а номер в тексте **не**
  становится ссылкой автоматически. Отсюда `copy_text` и `<code>`.
- **Перенос времени:** менеджер отвечает **прямо на карточку** новым временем,
  заявка ищется по `tgMessageId`. Формат `20.08 18:00-21:00`, год необязателен,
  `22:00-01:00` корректно уезжает за полночь. Ответ не про время игнорируется молча.
  Перенос на занятый слот отбивает exclusion-констрейнт, цена пересчитывается.
- **«Вернуть в работу»** (→ REVIEW) — страховка от случайного «Отменить». Может
  упереться в констрейнт: пока заявка была отменена, слот мог занять другой — тогда
  менеджер видит «слот уже занят», а не зависшую кнопку.
- **Безопасность вебхука** (адрес публичный): секретный заголовок
  `X-Telegram-Bot-Api-Secret-Token` + сверка chat_id с нашей группой. Проверено атаками:
  чужой секрет и чужой чат отбиваются.
- **Повторы:** 3 попытки с растущей паузой на сетевые сбои, 429 и 5xx; на 4xx не
  повторяем. Без этого моргнувшая сеть = неувиденная заявка.
- **Уведомление не роняет заявку:** ошибка Telegram только логируется.
- `Booking.tgMessageId` — карточка редактируется на месте, а не плодит сообщения.

## Правила бронирования — ПЕРЕСМОТРЕНЫ (решение заказчика)
Прежние ограничения выдумал разработчик, они мешали живым заявкам. Как сейчас:

- **Расписания нет.** Катер берут в любое время суток, на сутки и на несколько
  дней (максимум 7 — предохранитель от опечатки в датах). Окно «10:00–02:00» убрано.
- **Слот держит ТОЛЬКО подтверждённая бронь** (`BLOCKING_STATUSES = ['CONFIRMED']`).
  Раньше держали и NEW/REVIEW — необработанная заявка отбивала следующих клиентов.
  **Обратная сторона:** на один слот могут прийти несколько заявок, менеджер сам
  решает, кому отказать.
- **Буфера между бронями нет** (`BUFFER_MINUTES = 0`). Отказ — только при реальном
  нахлёсте. Вернуть паузу — одна строка в `booking-rules.ts`.
- Единственный содержательный отказ — `BOAT_BUSY`. Остальное отбивается лишь если
  физически бессмысленно: прошлое, меньше часа, больше 7 суток.

⚠️ Условие констрейнта в БД обязано совпадать с `BLOCKING_STATUSES`. Разъедутся —
код и база будут считать занятость по-разному. Миграция: `20260716140000_block_only_confirmed`.

В форме вместо «время до» — **выбор длительности** (часы, затем сутки): с «до»
пришлось бы гадать, какие это сутки при переходе через полночь.

## Готово — бронь от менеджера (/bron)
Клиенты звонят и пишут менеджеру лично, минуя сайт. Раньше такие брони нигде не
учитывались, и календарь занятости врал.

- **Запуск:** команда `/bron` или кнопка «➕ Создать бронь» под `/start`.
- **Шаги:** катер → начало (`ДД.ММ ЧЧ:ММ`) → длительность → имя → телефон → гости → комментарий.
- **Помечается** `source = MANUAL` и сразу `CONFIRMED`: менеджер уже поговорил с
  клиентом, подтверждать нечего — и слот сразу занят. В журнале записан автор.
- **Черновик — в БД** (таблица `BotDraft`), не в памяти: перезапуск PM2 обнулил бы
  наполовину введённую бронь. Заброшенные чистятся через 30 мин, иначе они
  перехватывали бы обычные сообщения в группе.
- Карточка в группе рисуется тем же кодом, что и заявка с сайта.

## Диалог /bron — кнопками, а не набором руками

Менеджер застрял на боевом: на шаг «когда начало?» прислал `17.07 18.00`, а
разбор требовал ТОЛЬКО двоеточие (`18:00`). Вернулся null, бот промолчал — в
группе он намеренно не отвечает на непонятное, чтобы не встревать в разговор
менеджеров, — и диалог встал намертво, без единого признака ошибки.

Лечение в двух местах сразу:

- **Шаги переведены на кнопки:** катер → день (14 кнопок: «Сегодня», «Завтра»,
  `19.07 вс`…) → час → минуты → длительность → гости. Набирать руками остаётся
  только имя и телефон — кнопками их не выбрать.
- **Разбор стал терпимее** там, где ввод всё же ручной («✍️ Другая дата», для
  дат дальше двух недель): разделитель времени — двоеточие, точка, дефис или
  пробел; минуты можно не писать (`17.07 18`); принимается и одна дата без
  времени — тогда время спросят кнопками.

Болтовня по-прежнему игнорируется: «привет», «да, 5 человек», `99.99 18:00`
разбор отвергает, и бот молчит.

При «катер занят» диалог возвращает к выбору дня кнопками, а не к набору руками:
катер, имя и телефон в черновике уже есть, менять надо только время.

**Перенос времени — теми же кнопками.** Кнопка «Изменить время» на карточке
раньше показывала подсказку «ответьте сообщением в формате ДД.ММ ЧЧ:ММ-ЧЧ:ММ» —
то есть ровно тот ручной ввод, на котором менеджер и застрял. Теперь она
запускает диалог: день → час → минуты → длительность (4 шага, шагов про клиента
у переноса нет). Ответ текстом на карточку продолжает работать как быстрый путь.

Черновик у обеих задач общий (`BotDraft`), различает их `mode: 'new' | 'retime'`:
шаги «день → час → минуты → длительность» одинаковые, а в конце — либо создание
брони, либо `applyReschedule` для той, чей id лежит в `bookingId`. Перенос идёт
через тот же `applyReschedule`, что и ответ текстом, и админка — иначе три
интерфейса разъехались бы.

### Панель под полем ввода: подписи зависят от режима приватности
`mainKeyboard()` в `lib/telegram.ts`, ставится на `/start` и `/help`, дальше
висит под полем ввода постоянно (`is_persistent`).

Нажатие кнопки reply-панели шлёт в чат ОБЫЧНЫЙ текст с её подписью. При
включённом режиме приватности бот такого текста не видит — до него доходят
только команды и ответы на его же сообщения, — и кнопка «➕ Создать бронь»
молча не работала бы. Тот же класс ошибки, что и застрявший диалог выше.

Поэтому подписи не зашиты, а выбираются по ответу Telegram: `getMe` отдаёт
`can_read_all_group_messages`, и бот сам решает — «➕ Создать бронь» / «❓ Помощь»
при выключенной приватности, голые `/bron` и `/help` при включённой. Ответ
кэшируется на 5 минут, так что переключение у @BotFather подхватится само, без
перезапуска. При ошибке запроса считаем приватность включённой: команды
работают всегда.

**Красивые подписи = выключить приватность** у @BotFather (`/setprivacy` →
Disable). Цена: бот начинает видеть всю переписку в группе.

⚠️ **Отсюда защита в вебхуке:** шаги диалога с текстом принимают ТОЛЬКО ответ на
сообщение бота (`reply_to_message.from.is_bot`). При включённой приватности иначе
и не бывает. Но если её выключить, бот начнёт видеть всё подряд — и без этой
проверки любая реплика менеджера на шаге «имя клиента» стала бы именем клиента.

## Готово — админка `/admin` (ТЗ, п. 6.3)
Вход, заявки, лодки, клиенты, сводка. Заказчик правит сайт сам, не трогая git.

**Вход — свой, не Supabase Auth и не Auth.js.** Supabase в проекте нет, а нужен один
сценарий «email + пароль» на двоих. `lib/auth.ts` на голом `node:crypto`: пароль —
`scrypt` (соль у каждого своя), сессия — подписанная HMAC httpOnly-кука. Тот же
подход, что и в `lib/google-sheets.ts`: сотня строк своего кода вместо зависимости
с чужим релиз-циклом (Auth.js v5 под Next 16 — beta).

- **Таблицы сессий нет** — админов двое, отзывать по одной некому. Разлогинить всех
  разом: сменить `ADMIN_SESSION_SECRET`, все подписи станут недействительны.
- **Админ перечитывается из БД на каждый запрос**, а не берётся из куки: иначе
  отключённый (`isActive = false`) ходил бы до истечения сессии.
- **Регистрации нет.** Админ заводится из консоли: `npx tsx scripts/admin.ts add <email> <пароль> <имя>`
  (ещё `passwd`, `off`, `list`). Открытая форма на публичном адресе означала бы, что
  аккаунт заведёт кто угодно.
- **Одинаковый ответ** на неверный email и неверный пароль, и пароль сверяется даже
  для несуществующего email (хеш-пустышка): разные тексты и разное время ответа
  выдали бы перебору список живых адресов. Плюс рейт-лимит 10 попыток / 10 мин на IP.

⚠️ **`proxy.ts` — это НЕ охрана.** Он лишь проверяет НАЛИЧИЕ куки, чтобы увести гостя
на форму входа: работает в edge-рантайме, где нет `node:crypto`, и подпись проверить
нечем. Настоящая охрана — `requireAdmin()` в каждой странице и **каждом server action**
(server action — открытая HTTP-ручка). Проверено: подделанная кука и кука с чужой
подписью отбиваются на 307 → `/admin/login`.

Файл называется `proxy.ts`, а не `middleware.ts`: в Next 16 старое имя устарело.

**Оркестрация вынесена в `lib/booking-workflow.ts`.** Смена статуса → перерисовка
карточки в Telegram → зеркало в Google Sheets. Раньше цепочка жила внутри вебхука;
теперь у брони два интерфейса, и бот с админкой ходят одним кодом. Разъедутся —
в группе повиснет «Новая» с живыми кнопками поверх отменённой заявки.

**Заявки** (`/admin/bookings`): список, фильтры (статус, лодка, источник, даты,
поиск по имени/телефону — в адресе, ссылку можно переслать), карточка, смена статуса,
перенос времени (длительность, не «до»), журнал, быстрый контакт.

⚠️ **Конкуренты за слот показываются в карточке** — это прямое следствие правила
«слот держит только CONFIRMED». На одно время лежат несколько заявок, и менеджер
выбирает, кому отказать; без этого блока он подтвердил бы одну и молча забыл остальные.
Кнопка «Подтвердить» умеет отвечать «катер уже занят» — пока менеджер думал, слот
мог занять другой через бота.

**Лодки** (`/admin/boats`): тексты RU/EN, цена, ТТХ, удобства, ярлык, флагман,
видимость, порядок. Удаления нет — на `Boat` висят брони, база не даст; только «скрыть».
Загрузки фото нет (решение заказчика): фото остаются в `public/boats/` в репозитории.
**Создания лодок тоже нет** — новая лодка заводится только через `lib/i18n.ts` +
`scripts/apply-fleet.ts`.

**Моменты** (`/admin/gallery`): загрузка фото и видео, скрыть/показать, порядок,
удаление. Единственный раздел с настоящей загрузкой файлов — подробности ниже.

**Клиенты** (`/admin/clients`): история обращений по телефону, признак повторного,
сумма подтверждённых. **Сводка** (`/admin`): необработанные, ближайшие прогулки,
конверсия по источникам за 30 дней (ТЗ, п. 6.6) — на базе `lib/analytics.ts`.

**`noindex` + `force-dynamic`** на всей админке: там телефоны и имена клиентов.

### ⚠️ Грабли: Prisma прячет код ошибки Postgres
`isExclusionViolation()` в `lib/bookings-db.ts` не ловил `23P01`, и подтверждение
брони на занятый слот падало **500-й вместо «катер занят»**. Внутри интерактивной
транзакции (`$transaction` с колбэком) Prisma 6.19 бросает
`PrismaClientUnknownRequestError` **вообще без поля `code`** — код остаётся только в
тексте ошибки. Проверки на `e.code === '23P01'` и на `P2010 + meta.code` обе мимо.
Теперь есть и третья — по тексту. Поймано живым прогоном админки; тест на подделанной
ошибке это скрыл бы.

### ISR включён (закрыт долг ТЗ, п. 7)
`export const revalidate = 300` на `/` и `/en` + `revalidatePath('/')` при сохранении
лодки. Без этого каталог собирался статически, флот замерзал на момент сборки, и
правка в админке не появилась бы на сайте до следующего деплоя — то есть раздел лодок
был бы бесполезен. В выводе сборки у `/` и `/en` теперь `Revalidate 5m`.

## Готово — галерея «Моменты» с загрузкой из админки
Раздел на главной перестал быть захардкоженным списком сгенерированных картинок:
таблица `GalleryItem`, содержимое правится в `/admin/gallery`. Первый раздел, где
файлы грузятся без git и пересборки.

**Файлы — в `uploads/gallery/` в корне проекта, вне git и вне `public/`.**
Выбрано вместо S3 Beget: ничего не надо подключать, а `deploy.sh` их не трогает,
потому что делает только `git checkout -- .` и `git pull`. ⚠️ **Не добавлять в
деплой `git clean -fdx`** — он снесёт все загрузки. Обратная сторона: загрузки не
попадают в бэкап через репозиторий, нужен отдельный бэкап папки.

⚠️ **Почему НЕ `public/` — на этом уже обожглись.** Next составляет список файлов
`public/` на этапе сборки, поэтому всё загруженное из админки после деплоя
отдавалось **404** до следующей пересборки: загрузка не работала вовсе. Поймали
не сразу, потому что первые файлы попали в папку до сборки и показывались
нормально. Теперь файлы лежат вне `public/` и раздаются обработчиком
`app/media/gallery/[file]` — URL вида `/media/gallery/<файл>`.

Обработчик держит **Range-запросы**: без них браузер не перематывает видео, а
Safari отказывается играть. Имя файла из URL проверяется на разделители и `..` —
иначе запрос вида `..%2F..%2F.env` вычитал бы что угодно с диска. Правку Nginx
сознательно не делали: она живёт вне git и потерялась бы при переезде сервера,
а обработчик работает одинаково на localhost и на боевом.

**Фото приводятся к webp** при загрузке (sharp, ширина до 1600, q80): снимок с
телефона — 8–12 МБ, без пережатия он уехал бы посетителям как есть. `rotate()`
разворачивает по EXIF, иначе портретные снимки лежат на боку; заодно EXIF с
геометками не попадает на сайт.

⚠️ **Перекодирование нужно не только ради размера — без него часть видео не
показывается.** Съёмка с iPhone приходит в **HEVC (h265)**, а Chrome и Firefox
не декодируют его в mp4: играет звук, картинки нет. Поймано на localhost, где
ffmpeg отсутствует и перекодирование пропускается; на боевом те же файлы уже
были переведены в H.264 и показывались нормально. Следствие в коде: **если
исходник не H.264, результат берётся даже когда он не стал меньше** — файл
поменьше, который не показывается, хуже файла побольше, который показывается.

**Звука на сайте нет вообще.** `-an` вырезает дорожку из файла (это и есть
экономия трафика), плюс `muted` на плеере — страховка для файла, не прошедшего
через ffmpeg. Для уже сжатых роликов есть `stripAudioInPlace`: снимает дорожку
через `-c:v copy`, без потери качества — гонять их второй раз через libx264
ради этого значило бы отнять качество впустую.

**Видео сжимается в фоне** (`lib/video.ts`, ffmpeg): 35,4 → 12,5 МБ на первых
пяти роликах, после снятия звука — 8,6 МБ.
- Работа уходит в `after()` из `next/server` — **уже после ответа**: ffmpeg тратит
  десятки секунд, столько server action не живёт, админка упёрлась бы в таймаут.
- ⚠️ **Файл заменяется на месте, под тем же именем.** Главная статична (ISR, 5 мин),
  и при смене имени она до ревалидации ссылалась бы на исчезнувший файл — видео
  пропало бы у посетителей. Контейнер сохраняется (mp4→mp4, mov→mov); webm не
  трогаем, VP9 пережимается слишком медленно.
- **H.264/mp4 намеренно, не VP9/webm**: mp4 играет везде, включая старые iPhone.
  `+faststart` — иначе браузер не начнёт воспроизведение до полной загрузки.
- Режим отказа безопасный: нет ffmpeg, ошибка или результат не меньше — остаётся
  оригинал, временный файл убирается.

⚠️ **Локальная разработка без ffmpeg.** На Windows его нет, поэтому загруженные
на localhost ролики остаются в исходном виде — и если это HEVC с iPhone, картинки
не будет (см. выше). Боевого это не касается. Нужна одинаковая картина — поставить
ffmpeg и прогнать `npx tsx scripts/compress-videos.ts`, он идемпотентный.

**`serverActions.bodySizeLimit` = 64 МБ** (по умолчанию 1 МБ — видео с телефона
молча не проходило бы). Держать в согласии с `MAX_FILE_BYTES` в `lib/gallery.ts`.

**`sharp` закреплён в зависимостях явно** — приходил транзитивно от Next и мог
пропасть при переустановке, уронив загрузку.

Пустая галерея скрывает раздел на сайте целиком — плашка без содержимого
выглядела бы поломкой.

### ⚠️ Грабли: grid-аккордеон из FAQ не годится для разделов главной
Выпадающие разделы (`components/nevsky/collapsible-section.tsx`) сделаны на
измеряемом `max-height` с `ResizeObserver`, а не приёмом `grid-rows-[0fr] → [1fr]`,
как аккордеон в FAQ. Причина: схлопывание требует `overflow-hidden` на содержимом,
но он же обнуляет дорожку `1fr` на сложном контенте (вложенные сетки, `.reveal`) —
раздел открывался в нулевую высоту. **Не «упрощать» обратно к grid-приёму**,
выглядит как дублирующий код, но им не является.

### Что проверено вживую (админка)
Гость → редирект на вход; неверный пароль отбит; вход; подделанная кука и кука с
чужой подписью → 307 на логин; выход → панель снова закрыта. Список, счётчики,

`REPEAT`). Подтверждение заявки; подтверждение конкурента на тот же слот →
«катер уже занят подтверждённой бронью» вместо 500. Правка цены в админке доехала
до публичной страницы. `npx tsc --noEmit` и `npm run build` чисты.
