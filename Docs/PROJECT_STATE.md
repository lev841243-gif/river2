# Состояние проекта — «Судоходная Компания Дно»

Краткий хэндовер, чтобы продолжить разработку в любом режиме без потери контекста.

## Что это
Сайт аренды катеров в Санкт-Петербурге. Next.js 16 (App Router), React 19, TypeScript, Tailwind v4.
Двуязычный: RU на `/`, EN на `/en` (словарь — `lib/i18n.ts`). Домен заказчика: prokatkaterov.ru.
Бренд в интерфейсе: «Судоходная Компания Дно».

## Стек (согласован)
- Приложение: Next.js на **Beget** (VPS/Node, PM2 + Nginx).
- БД: **PostgreSQL** (Beget Cloud) + **Prisma** ORM.
- Авторизация админки: своя (Auth.js/сессия) — этап админки.
- Фото лодок: WebP в `public/boats/<папка>` (позже — `/uploads` на сервере).
- Telegram (grammY) и Google Sheets — этапы уведомлений/отчётности.
- Хостинг и деплой — см. `Docs/DEPLOY_BEGET.md`.

## Готово (этап 1 — бронирование на сайте с календарём занятости)
Клиент бронирует полностью на сайте: катер → календарь → время «с»/«до» → контакты. Занятость проверяется в реальном времени.

**Правила бронирования — `lib/booking-rules.ts` (единый источник для клиента и сервера):**
окно 10:00–02:00 (`DAY_START_HOUR`/`DAY_END_HOUR`), минимум 2 ч, максимум 12 ч, буфер между бронями 60 мин, горизонт 180 дней, шаг сетки 30 мин, слот держат статусы NEW/REVIEW/CONFIRMED. Меняются константами в одном файле.

**Время.** `lib/spb-time.ts` — все даты считаются явно в петербургском поясе (UTC+3, фикс. с 2014 г.), а не в локальном поясе браузера. Иначе гость из другого пояса на EN-версии забронировал бы не то время. Проверено под TZ Москвы/Нью-Йорка/Токио — результат одинаковый.

**Защита от двойного бронирования — на уровне БД.** Миграция `20260715183600_booking_overlap_guard`: exclusion-констрейнт `EXCLUDE USING gist` по `(boatId, tstzrange(startAt, endAt))` для блокирующих статусов. Проверка «посмотрели → вставили» не спасает от гонки двух одновременных заявок; констрейнт делает пересечение физически невозможным. Postgres отдаёт код `23P01`, `lib/bookings-db.ts` ловит его и превращает в понятный 409 `BOAT_BUSY`. Требует `btree_gist` (миграция ставит сама).

**Схема.** `startAt`/`endAt` — `@db.Timestamptz(3)`, `endAt` обязателен. Оба обязательны для констрейнта: от `timestamp` без таймзоны `tstzrange` зависит от сессионной TimeZone и не годится в индексное выражение. Прочие DateTime тоже переведены в timestamptz (как в ТЗ).

**Файлы:**
- `lib/booking-rules.ts` — правила + Zod-схема заявки + honeypot.
- `lib/spb-time.ts` — петербургское время.
- `lib/booking-slots.ts` — расчёт свободных слотов (общий для календаря и сервера).
- `lib/bookings-db.ts` — занятость, создание заявки, определение источника по UTM/referer, клиент по телефону.
- `lib/rate-limit.ts` — лимит частоты.
- `app/api/availability/route.ts` — `GET ?boatSlug&from&to` → занятые интервалы.
- `app/api/bookings/route.ts` — `POST`, Zod + honeypot + рейт-лимит (5/10 мин) + проверка занятости.
- `components/nevsky/booking-calendar.tsx` — календарь, занятые дни недоступны.
- `components/nevsky/booking-modal.tsx` — шаги «когда» → «контакты» → успех.
- `components/nevsky/booking-context.tsx` — `BookingProvider`/`useBooking`.
- `lib/utm.ts` + `components/nevsky/utm-capture.tsx` — UTM → cookie `utm_data` (1 год) → в заявку.

**Важно про `boatId`.** В UI лодка — это `slug` (`getBoats()` мапит `id: r.slug`), а `Booking.boatId` — FK на `Boat.id` (uuid). API принимает slug и резолвит его сам. Не слать uuid с клиента.

**Связь с менеджером.** Telegram/WhatsApp/звонок — только контакт, не канал брони (`contacts` в `lib/i18n.ts`, блок в `Cta`).

**Что проверено вживую** (локальная БД `river2_dev`): констрейнт отбивает пересечение и пропускает встык/отменённые; POST даёт 201, повтор — 409; буфер, минимум 2 ч, прошедшее время, часы работы, рейт-лимит, honeypot (молча 201 без записи), неизвестная лодка — 404; заявка через UI легла в БД с верным временем МСК, ценой (3 ч × 50 000 = 150 000) и историей статусов; календарь на RU и EN показывает занятость. `npx tsc --noEmit` и `npm run build` чисты.

## Готово — Telegram-уведомления
Заявка приходит в группу Telegram карточкой с кнопками. Пробел «менеджер не видит
заявок» закрыт.

- **Бот:** @river_dno_bot. **Группа:** «neva», chat_id `-5416776321`.
- **Файлы:** `lib/telegram.ts` (API + вёрстка сообщения), `app/api/telegram/webhook/route.ts` (нажатия).
- **Кнопки:** WhatsApp и Telegram — прямые ссылки; «📋 Номер» копирует телефон;
  «Подтвердить» / «Отменить» / «Изменить время».
- **Почему звонок через копирование:** проверено на живом боте — `tel:` в кнопке
  Telegram **отклоняет** («Wrong port number specified»), а номер в тексте **не**
  становится ссылкой автоматически. Отсюда `copy_text` и `<code>`.
- **Перенос времени:** ответ на подсказку (force_reply), id заявки спрятан в её тексте —
  состояние диалога хранить не нужно. Формат `20.08 18:00-21:00`, год необязателен,
  `22:00-01:00` корректно уезжает за полночь. Перенос на занятый слот отбивает
  exclusion-констрейнт, цена пересчитывается.
- **Безопасность вебхука** (адрес публичный): секретный заголовок
  `X-Telegram-Bot-Api-Secret-Token` + сверка chat_id с нашей группой. Проверено атаками:
  чужой секрет и чужой чат отбиваются.
- **Уведомление не роняет заявку:** ошибка Telegram только логируется.
- `Booking.tgMessageId` — карточка редактируется на месте, а не плодит сообщения.

**Проверено на боевом:** заявка → уведомление в группу → нажатие «Подтвердить»
(в журнале записан автор) → перенос времени на 25.08 с пересчётом цены.
