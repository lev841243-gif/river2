## 4. Ключевые решения и почему именно так

### Защита от двойного бронирования — в БД, а не в коде
Exclusion-констрейнт Postgres `EXCLUDE USING gist`. Проверка «посмотрели занятость
→ вставили» не спасает от гонки двух одновременных заявок: обе увидят «свободно».
Констрейнт делает пересечение физически невозможным. Postgres отдаёт `23P01`,
код превращает его в понятный 409 `BOAT_BUSY`.

⚠️ **Условие констрейнта обязано совпадать с `BLOCKING_STATUSES`** в
`lib/booking-rules.ts`. Разъедутся — код и база будут считать занятость по-разному.

Требует расширение `btree_gist`. На PG13+ оно *trusted* и ставится владельцем БД
без суперюзера — проверено на боевой (`cloud_user`, `rolsuper = f`).

### Время — всегда московское, явно
`lib/spb-time.ts`. Даты считаются в UTC+3 явно, не через локальное время браузера.
Иначе гость из другого пояса на EN-версии забронировал бы не то время. Проверено
под TZ Москвы / Нью-Йорка / Токио.

`startAt`/`endAt` — `@db.Timestamptz(3)`. Это не косметика: от `timestamp` без
таймзоны `tstzrange` зависит от сессионной TimeZone и не годится в индексное
выражение — констрейнт просто не создастся.

### Правила брони пересмотрены заказчиком (было → стало)
Прежние ограничения выдумал разработчик, они мешали живым заявкам.

| | Было | Стало |
|---|---|---|
| Часы | 10:00–02:00 | **нет расписания** |
| Длительность | 2–12 ч | 1 ч – 7 суток |
| Слот держат | NEW + REVIEW + CONFIRMED | **только CONFIRMED** |
| Буфер между бронями | 60 мин | **0** |

**Обратная сторона:** на один слот могут прийти несколько заявок — менеджер сам
решает, кому отказать. Это цена за то, чтобы необработанная заявка не блокировала катер.

В форме «время до» заменено на **длительность** — с «до» пришлось бы гадать, какие
это сутки при переходе через полночь.

### Фото лодок — в репозитории
`public/boats/<папка>/`. В БД только имена файлов. Значит, **добавить лодку с фото
можно только через git + пересборку**. Заказчик решил пока так и оставить.
Когда админка потребует загрузку — выбирать между `/uploads` на сервере и
объектным хранилищем Beget (S3, есть в панели).

## 5. Грабли — сюда смотреть первым делом

### Prisma: `migrate deploy` не обновляет клиент
Миграция добавляет столбец в **базу**, но не в сгенерированный **клиент** в
`node_modules`. Приложение соберётся и упадёт в рантайме:
`Unknown argument 'tgMessageId'`. Обычно спасает `postinstall` при `npm ci`, но
если зависимости не менялись — `npm ci` пропускают, и клиент остаётся старым.
**Менялась `schema.prisma` → `npx prisma generate`.**

### Turbopack не пересобирает CSS
После правки `globals.css` dev-сервер продолжает отдавать старые стили. Проверять
надо не файл на диске, а то, что реально отдаёт сервер:
```bash
CSS=$(curl -s http://localhost:3000/ | grep -oE '/_next/static/[^"]*\.css' | head -1)
curl -s "http://localhost:3000$CSS" | grep -c 'имя-анимации'
```
Лечится только перезапуском с `rm -rf .next`.

### Telegram: `tel:` в кнопке отклоняется
`inline keyboard button URL 'tel:+7...' is invalid: Wrong port number specified`.
Номер в тексте тоже **не** становится ссылкой автоматически. Поэтому звонок сделан
через `copy_text` и `<code>` (тап копирует), а WhatsApp/Telegram — обычными url.

### Telegram: `reply_to_message` не вкладывается рекурсивно
Документированное поведение: объект в этом поле **не содержит собственного**
`reply_to_message`. Код, лезущий на два уровня вглубь, в бою не работает. Тест на
подделанной структуре это скроет — проверять только на реальном формате.

### Telegram: `force_reply` + `selective: true` целится в автора исходного сообщения
Если бот отвечает на своё же сообщение — форма ответа не откроется ни у кого.

### Telegram: косметика не должна блокировать действие
`answerCallbackQuery` (гасит «часики») жил **перед** самим действием, и его падение
обрывало обработчик — кнопка не делала ничего. Callback живёт ~минуту: хватает
моргнувшей сети. Теперь ошибки проглатываются.

### Nginx: `$proxy_add_x_forwarded_for` — дыра в рейт-лимите
Он **дописывает** к заголовку от клиента, а код берёт первый адрес — то есть
выдуманный клиентом. Рейт-лимит обходится подстановкой фальшивого IP.
Проверено атакой на живом сервере. Правильно: `proxy_set_header X-Forwarded-For $remote_addr;`

### Honeypot: не отвечать ошибкой
Zod-правило `max(0)` отбивало поле раньше проверки и возвращало боту `400` с
подсказкой, какое поле его выдало. Ловушка должна молча принять и выбросить (`201`).

### Windows: регистр в именах файлов
`hero.mp4` и `Hero.mp4` — один и тот же файл. Так чуть не потерялся исходник видео.

### Консоль Windows ломает кириллицу
`psql` и `curl` с кириллицей в аргументах — битая кодировка (`strings must be
encoded in UTF-8`). Обходить через Python-скрипт с `PGCLIENTENCODING=UTF8`.
На приложение не влияет: там Node и UTF-8 везде.

## 7. Технический долг

1. **Google Sheets не проверен на живой таблице** — нет доступов. Код написан без
   `googleapis` (десятки мегабайт ради одного `append`), на `fetch` + `crypto`.
   Подпись JWT проверена тестом.
2. **Google Calendar** — заказчик просил «таблица и/или календарь». Сделана
   таблица. Календарь — тот же сервисный аккаунт.
3. ~~**ISR для каталога**~~ → **закрыто на этапе 2**: `revalidate = 300` на `/` и `/en`
   плюс `revalidatePath` при сохранении лодки в админке. Подробности —
   в `PROJECT_STATE.md`.
4. **Рейт-лимит в памяти процесса** — обнуляется при рестарте, не общий для
   кластера PM2. Больше одного инстанса → нужен Redis.
5. **Мобильная полировка**: зоны нажатия мельче 44px (RU/EN — 37×24, значки
   соцсетей — 36×36), 11 элементов с текстом мельче 12px. Горизонтальной
   прокрутки нет — вёрстка адаптивная и здоровая, отдельная мобильная версия НЕ нужна.
6. **Видео в Hero откачено** — автозапуск на iOS не поборол (вероятно, режим
   энергосбережения). Файлы `public/images/hero.mp4` (1.3 МБ, пережат с 4.9 МБ,
   SSIM 0.969) и `hero-poster.webp` лежат, но не подключены.
7. **Оригинал видео 4.8 МБ остался в истории git** — утёк через `git add -A`.
   Чистится только переписыванием истории с force-push.
8. **`hero-neva.png` (1.8 МБ)** — используется как фон Hero.
9. ~~**`tsconfig.tsbuildinfo`** закоммичен~~ → **убран на этапе 2**. Оказался не
   косметикой: на сервере его меняет каждый `npm run build`, после чего `git pull`
   отказывался обновлять ветку и деплой тихо не доезжал.
